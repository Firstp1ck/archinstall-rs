# EFISTUB Bootloader - Issues & Recommended Fixes

Analysis of the EFISTUB implementation in `src/core/services/bootloader.rs` (match arm `2`, lines 316-371)
and related files (`sysconfig.rs`, `system.rs`, `flow.rs`).

---

## Critical Issues

### 1. Missing CPU microcode initrd in efibootmgr entries and startup.nsh

**File:** `src/core/services/bootloader.rs:354-369` (non-UKI path)

The pacstrap plan (`system.rs:229-238`) correctly installs `intel-ucode` or `amd-ucode`,
but the EFISTUB boot entries **never load the microcode**:

```bash
# Current (broken) — no microcode:
efibootmgr --create ... --loader '\\vmlinuz-linux' \
  --unicode "$OPTS initrd=\\initramfs-linux.img"

# startup.nsh also missing microcode:
vmlinuz-linux $OPTS initrd=\initramfs-linux.img
```

The Linux EFISTUB processes `initrd=` parameters sequentially. Without the microcode
initrd, CPU errata fixes and security mitigations are not applied at boot.

**Fix:** Detect installed microcode package and prepend its initrd:

```bash
# efibootmgr entry:
UCODE=""; [ -f /boot/intel-ucode.img ] && UCODE="initrd=\\\\intel-ucode.img ";
[ -f /boot/amd-ucode.img ] && UCODE="initrd=\\\\amd-ucode.img ";
efibootmgr --create ... --loader '\\vmlinuz-linux' \
  --unicode "$OPTS ${UCODE}initrd=\\initramfs-linux.img"

# startup.nsh:
vmlinuz-linux $OPTS initrd=\intel-ucode.img initrd=\initramfs-linux.img
```

**Impact:** High — security & stability (but system still boots without it).

**Note:** systemd-boot entries (`arch.conf`) have the same missing-microcode bug
at lines 281-286. UKI mode is unaffected because mkinitcpio embeds microcode
into the unified image.

### 2. No UEFI fallback path (`EFI/BOOT/BOOTX64.EFI`) for non-UKI EFISTUB

**File:** `src/core/services/bootloader.rs:353-369`

UKI EFISTUB (line 322-326) and Limine (line 196-197) both copy their EFI binary to
the UEFI standard fallback path `EFI/BOOT/BOOTX64.EFI`. Non-UKI EFISTUB does **not**.

The only fallback for non-UKI is `startup.nsh`, which requires:
- The firmware to include a UEFI Shell
- The UEFI Shell to auto-execute `startup.nsh` from the ESP root

Many consumer motherboards and VMs do **not** ship a UEFI Shell. If the NVRAM entries
are lost (firmware update, CMOS reset, VM snapshot restore), the system will not boot.

**Fix:** Copy `vmlinuz-linux` to `EFI/BOOT/BOOTX64.EFI` and embed a minimal
startup.nsh. However, this has limited utility because `BOOTX64.EFI` as a raw
kernel still needs cmdline parameters, which UEFI firmware does not provide when
falling back.

**Better fix:** Recommend UKI mode when EFISTUB is selected (UKI is self-contained
and works at the fallback path). Consider showing a warning in the TUI info panel
or validation step when EFISTUB is chosen without UKI.

**Impact:** Critical on affected firmware — system will not boot if NVRAM is lost.

### 3. No pacman hook to update startup.nsh or efibootmgr on kernel upgrades (non-UKI)

**File:** `src/core/services/bootloader.rs:353-369`

UKI EFISTUB installs `99-efistub-uki-fallback.hook` (line 327-339), and Limine installs
`99-limine.hook` (line 200-211). Non-UKI EFISTUB installs **no hook**.

When the `linux` package is upgraded:
- `vmlinuz-linux` is replaced on disk (efibootmgr entry still points to it — fine)
- `initramfs-linux.img` is regenerated by mkinitcpio (fine)
- But `startup.nsh` still contains the **old cmdline** from install time

If the root UUID ever changes (e.g., partition resize, btrfs balance), startup.nsh
will have a stale `root=UUID=...` value.

**Fix:** Add a pacman hook triggered by `linux` package upgrades that regenerates
`startup.nsh`:

```ini
[Trigger]
Operation = Install
Operation = Upgrade
Type = Package
Target = linux

[Action]
Description = Refresh EFISTUB startup.nsh after kernel upgrade
When = PostTransaction
Exec = /bin/sh -c "... regenerate startup.nsh ..."
```

**Impact:** Medium — only matters if root UUID changes after install or startup.nsh
is the active boot method.

## Moderate Issues

### 4. No duplicate efibootmgr entry cleanup

**File:** `src/core/services/bootloader.rs:341-351, 357-368`

`efibootmgr --create` always adds a **new** NVRAM entry. If the installer is run
multiple times, or the user re-installs, duplicate "Arch Linux" entries accumulate.
Some firmware have limited NVRAM space (as low as 4 KiB) and may fail silently
when full.

**Fix:** Before `--create`, delete existing entries with the same label:

```bash
# Remove existing entries with matching label
for bootnum in $(efibootmgr | grep -i 'Arch Linux' | grep -oP 'Boot\K[0-9A-Fa-f]{4}'); do
  efibootmgr -b "$bootnum" -B 2>/dev/null || true
done
```

**Impact:** Medium — clutters firmware boot menu; can fill NVRAM on repeated installs.

---

### 5. No boot order management after entry creation

**File:** `src/core/services/bootloader.rs:341-351, 357-368`

After `efibootmgr --create`, the new entry is typically prepended to the boot order.
But this is firmware-dependent. Some firmware may not update `BootOrder` at all,
leaving the new entry present but never selected.

**Fix:** Explicitly set boot order after creating entries:

```bash
NEWBOOT=$(efibootmgr | grep 'Arch Linux' | head -1 | grep -oP 'Boot\K[0-9A-Fa-f]{4}')
efibootmgr -o "$NEWBOOT" || true
```

Or use `efibootmgr --create` with `--active` (it's the default, but being explicit
doesn't hurt).

**Impact:** Low-Medium — firmware-specific; most UEFI implementations handle this.

### 6. startup.nsh missing filesystem device prefix

**File:** `src/core/services/bootloader.rs:354-356`

Current startup.nsh content:
```
vmlinuz-linux root=UUID=... rw initrd=\initramfs-linux.img
```

Some UEFI Shell environments require an explicit filesystem mapping prefix:
```
FS0:\vmlinuz-linux root=UUID=... rw initrd=FS0:\initramfs-linux.img
```

Without it, the UEFI Shell may not find the kernel if the current working directory
is not the ESP root.

**Fix:** Auto-detect or hardcode the FS mapping:

```bash
cat > /boot/startup.nsh <<'NSHEOF'
@echo -off
for %a run (0 9)
  if exist FS%a:\vmlinuz-linux then
    FS%a:\vmlinuz-linux root=UUID=... rw initrd=FS%a:\initramfs-linux.img
  endif
endfor
NSHEOF
```

Or use a simpler approach — prepend `FS0:` (the ESP is almost always FS0).

**Impact:** Low-Medium — depends on firmware UEFI Shell behavior; many auto-set CWD
to the ESP.

### 7. Double `bash -lc` nesting in systemd-boot loader.conf

**File:** `src/core/services/bootloader.rs:269-271`

This is not EFISTUB-specific, but worth noting:

```rust
cmds.push(chroot_cmd(
    "bash -lc 'cat > /boot/loader/loader.conf <<EOF\n...\nEOF'",
));
```

`chroot_cmd()` already wraps in `arch-chroot /mnt bash -lc '...'`, so this creates:
```
arch-chroot /mnt bash -lc 'bash -lc '\''cat > /boot/loader/loader.conf ...'\'''
```

The double nesting is unnecessary and may cause quoting issues.

**Fix:** Remove the inner `bash -lc`:
```rust
cmds.push(chroot_cmd(
    "cat > /boot/loader/loader.conf <<EOF\n...\nEOF",
));
```

## Minor Issues / Hardening

### 8. Hardcoded kernel name `linux` — no multi-kernel support

**File:** `src/core/services/bootloader.rs:346-365`

All EFISTUB entries reference `vmlinuz-linux` and `initramfs-linux.img`. If the user
selects `linux-lts`, `linux-zen`, or `linux-hardened` as their kernel
(`state.selected_kernels`), no efibootmgr entries are created for those kernels.

For UKI, the preset path is also hardcoded to `linux.preset` and
`arch-linux.efi` / `arch-linux-fallback.efi`.

**Fix:** Iterate over `state.selected_kernels` and create entries for each:
- `vmlinuz-linux-lts` / `initramfs-linux-lts.img` / `arch-linux-lts.efi`
- etc.

**Impact:** Medium for users who select alternative kernels (common use case).

---

### 9. UKI fallback hook only copies default, not fallback UKI

**File:** `src/core/services/bootloader.rs:338`

The `99-efistub-uki-fallback.hook` copies only `arch-linux.efi` to
`EFI/BOOT/BOOTX64.EFI`. If the default UKI is broken (e.g., bad initramfs),
the fallback UKI (`arch-linux-fallback.efi`) is available in
`EFI/Linux/` but not at the UEFI standard fallback path.

**Fix:** Copy whichever UKI exists, preferring fallback as the safety net:

```bash
Exec = /bin/sh -c "for f in /boot/EFI/Linux/arch-linux-fallback.efi \
  /boot/EFI/Linux/arch-linux.efi; do \
  [ -f \"$f\" ] && /usr/bin/install -Dm0644 \"$f\" /boot/EFI/BOOT/BOOTX64.EFI && break; done"
```

Or copy default to BOOTX64.EFI (current behavior) and accept the risk, since the
fallback is still reachable via the efibootmgr NVRAM entry.

**Impact:** Low — edge case where default UKI is broken but fallback works.

---

### 10. `efibootmgr` error handling uses `||` chaining that swallows the second entry's status

**File:** `src/core/services/bootloader.rs:346-351`

```bash
efibootmgr --create ... --label 'Arch Linux' ... || echo "WARNING: ...";
efibootmgr --create ... --label 'Arch Linux (fallback UKI)' ... || echo "WARNING: ...";
efibootmgr --verbose || true;
```

The `;` after the first `|| echo` means the second `efibootmgr --create` always runs
(which is correct). But if the first fails, the `echo` sets `$?` to 0, and the `fi`
at the end of the `if` block succeeds. This is actually fine — both commands are
independent and both have their own `|| echo` fallback.

However, if `mountpoint` and `mount` both fail (no efivarfs support at all), the
entire `if` block is skipped silently. Consider logging this case.

**Impact:** Low — informational.
